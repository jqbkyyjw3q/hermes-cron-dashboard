# Hermes Cron Dashboard 信息架构与功能设计

> **目的：** 先把功能与页面结构设计清楚，再进入视觉与实现。

## 一、当前问题

用户当前反馈的核心不是“少一个功能”，而是“信息组织方式不对”。

当前版本的主要问题：
1. **所有内容铺在一个页面**，导致首页承担了过多职责。
2. **首页既像看板、又像搜索页、又像归档页、又像详情页入口页**，信息角色冲突。
3. **详情虽然已做成独立页视图，但整体导航模型仍像单页堆叠页面**。
4. **搜索、分类、历史、今日内容没有形成清晰的信息流转路径**。
5. **页面优先级不清**：用户想看“今天发生了什么”“某类内容有哪些”“某条内容全文是什么”“历史上同类内容怎么变化”，但现在这些目标混在一起。

## 二、设计目标

新的版本应该满足这 5 个目标：

1. **首页只做总览，不做全量承载。**
2. **列表页负责浏览，详情页负责阅读。**
3. **搜索页独立出来，成为完整功能页面。**
4. **历史归档独立成列表体系，不与今日内容混排。**
5. **路由清晰、可收藏、可分享、可回退。**

## 三、推荐信息架构（多页面）

建议改成下面这套 5 页面结构：

### 1. 首页 `/`
**定位：今日总览页 / Dashboard**

**只放：**
- 顶部全局导航
- 今日摘要（今天多少任务、多少分类、最近更新时间）
- 今日头条（只展示今天最重要的 3~6 条）
- 今日分类概览（每类 1~3 条）
- 快速入口：
  - 查看全部今日内容
  - 查看历史归档
  - 进入全文搜索

**首页不要放：**
- 大量历史卡片
- 搜索结果列表
- 过多详情摘要
- 全量分类长列表

**首页目标：**
让用户 10 秒内知道“今天有哪些重点”“接下来该点哪里”。

---

### 2. 今日内容列表页 `/today`
**定位：今天的完整内容列表页**

**展示方式：**
- 按分类分组
- 每类显示今天所有最新内容
- 支持：
  - 分类筛选
  - 简单搜索（仅限今日）
  - 排序（默认按分类 + 时间）

**每张卡片显示：**
- 标题
- 来源任务
- 内容日期
- 简短摘要
- 进入详情页

**目标：**
把“今天所有结果”集中到一页，不再塞在首页。

---

### 3. 全文搜索页 `/search`
**定位：独立搜索工作台**

**展示内容：**
- 顶部搜索框
- 搜索说明
- 搜索结果列表
- 结果统计（命中数、分类分布）
- 支持筛选：
  - 分类
  - 时间范围（今天 / 近7天 / 全部）
  - 来源类型（当前内容 / 历史归档）

**每条结果展示：**
- 标题
- 命中的上下文摘要
- 关键词高亮
- 来源任务
- 内容日期
- 进入详情页

**为什么搜索页必须独立：**
搜索是一个完整任务流，不应和首页总览混在一起。

---

### 4. 历史归档页 `/archive`
**定位：历史浏览页**

**展示内容：**
- 日期维度浏览
- 可按日期 / 分类切换
- 支持分页或按月分组
- 支持按任务名筛选

**推荐结构：**
- 左侧：日期/月份导航
- 右侧：对应日期的归档列表

**目标：**
解决“历史内容很多，但不该和今天内容铺在一起”的问题。

---

### 5. 详情页 `/item/:id`
**定位：单篇内容阅读页**

**详情页结构建议：**
- 顶部返回导航（返回今天 / 返回搜索结果 / 返回归档）
- 主标题
- 元信息：来源任务、分类、日期、调度信息
- 正文 Markdown 渲染区
- 相关推荐：
  - 同分类上一篇 / 下一篇
  - 同任务历史记录

**详情页只做一件事：**
让用户安静阅读全文。

## 四、推荐导航结构

建议顶部主导航固定为：

- **首页**
- **今日内容**
- **搜索**
- **历史归档**

详情页不放在主导航，而是通过列表页进入。

## 五、推荐用户路径

### 路径 A：看今天重点
`首页 -> 今日头条 -> 详情页`

### 路径 B：按分类浏览今天内容
`首页 / 今日内容页 -> 分类 -> 详情页`

### 路径 C：查关键词
`搜索页 -> 搜索结果 -> 详情页`

### 路径 D：回看历史
`历史归档页 -> 某天 / 某类 -> 详情页`

这样每条路径都清楚，不会再互相打架。

## 六、页面职责边界

### 首页
- 负责“告诉用户今天有什么”
- 不负责“展示所有内容”

### 今日内容页
- 负责“浏览今天的完整清单”
- 不负责“全文搜索”

### 搜索页
- 负责“查找”
- 不负责“首页摘要”

### 历史归档页
- 负责“回看过去”
- 不负责“今天的重点推荐”

### 详情页
- 负责“阅读正文”
- 不负责“承担聚合列表功能”

## 七、数据层对应关系

结合当前 `scripts/build_data.py` 里已有数据，功能设计可以直接映射：

### 已有数据
- `today_headlines`
- `current`
- `history`
- `stats`

### 对应页面建议
- 首页 `/` 使用：`stats + today_headlines + current(截取)`
- 今日内容页 `/today` 使用：`current`
- 搜索页 `/search` 使用：`current + history`
- 历史归档页 `/archive` 使用：`history`
- 详情页 `/item/:id` 使用：`current + history` 按 id 查询

所以目前数据层**足够支撑多页面拆分**，重点是前端结构重构，不是先改后端。

## 八、推荐第一阶段功能范围（先做对）

建议先做一个 **MVP 多页面版**，不要一上来做太复杂：

### Phase 1
1. 首页 `/`
2. 今日内容页 `/today`
3. 搜索页 `/search`
4. 历史归档页 `/archive`
5. 详情页 `/item/:id`

### Phase 1 暂时不做
- 登录态
- 收藏
- 复杂统计图表
- 多维高级筛选器
- 深度联动动画

先把信息结构做对。

## 九、页面视觉建议（配合“素雅”方向）

### 整体原则
- 以**文档阅读感**为主，不要过度 dashboard 化
- 首页更像“主编页”
- 列表页更像“目录页”
- 详情页更像“正文阅读页”

### 视觉控制建议
- 少用大面积深色块
- 弱化卡片感
- 增加留白和分组层级
- 保持极少数强调色
- 搜索页和归档页以“工具性清晰”为主

## 十、推荐实现顺序

### Step 1
先改路由与页面拆分：
- `/`
- `/today`
- `/search`
- `/archive`
- `/item/:id`

### Step 2
把首页内容收缩成总览型

### Step 3
把搜索独立成完整页面

### Step 4
把历史归档独立成完整页面

### Step 5
最后再统一视觉语言

## 十一、最终建议结论

**结论：**
你指出的问题是对的，当前不应该再继续在一个页面里打补丁，而应该直接切成多页面的信息架构。

### 推荐最终方案
- 首页：只做总览
- 今日内容：只做今天的完整浏览
- 搜索：独立页
- 历史归档：独立页
- 详情：独立页

这是后续继续优化 UI、搜索、归档体验的正确基础。

---

## 十二、下一步可直接执行的实现计划

如果确认按这个方案走，下一步就按下面做：

1. 重构 `index.html` + `assets/app.js`
2. 引入前端路由状态（或简易 hash/query 路由）
3. 拆分为 5 个页面视图
4. 首页只保留摘要模块
5. 搜索和归档迁出首页
6. 最后统一视觉样式

这个方向比继续在单页上做修补更正确，也更耐用。
